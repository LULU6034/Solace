# Voice Design — Sonder 语音聊天系统

## 概述

Sonder（命名源自"意识到每个路人都有和你一样复杂而鲜活的人生"）— 侧边栏语音聊天页面，固定单 Agent 语音对话。支持流式 TTS、语音打断、情感表达与共情、用户情绪检测，以及四层记忆系统（短期/中期/情景/长期）。

### 核心设计哲学

- **从"记住"到"理解"** — 记忆不仅是事实存储，更是关系网络和自我反思
- **从"表达"到"共鸣"** — 情感不仅是标签映射，更是多维度理解和人格化回应
- **从"问答"到"对话"** — 交互不仅是回合制，而是流畅自然的实时交流

### 核心约束

- **语音范围**: 仅在侧边栏语音页面内使用，不做全局唤醒词/桌面常驻监听
- **Agent 数量**: 固定一个 Agent 拥有语音能力，可调用所有工具
- **对话模式**: 半双工 + 语音打断 + 按键打断（丢弃重新开始）
- **TTS 引擎**: CosyVoice2-0.5B，CPU 推理，1-2s/句
- **GPU 约束**: RTX 3050 Laptop 4GB 显存，无法运行完整版 CosyVoice 2
- **隐私**: 混合模式 — TTS/STT 本地处理，LLM 走云端

### 通信与进程管理

**音频传输优化:**
- TTS 音频通过 Node.js → Electron 主进程 → renderer 的 IPC 通道传输
- 使用二进制 WebSocket（`ws` 支持 `send(Buffer)`）传输音频块，避免 Base64 膨胀
- 前端 Web Audio API `decodeAudioData()` 直接解码接收到的 ArrayBuffer
- 音频格式从 CosyVoice 原始 PCM/WAV 转换为 Opus 以减少带宽（~10x 压缩）

**进程生命周期:**
- Electron 启动时预加载 Python 服务（模型加载 5-10s），保持长连接复用
- 封装为 `child_process.spawn()`，注册 `app.on('will-quit')` + `process.on('exit')` 确保 Python 进程被 SIGTERM
- 主进程退出时强制 kill（`tree-kill` 确保子进程树完全清理，无残留）
- 健康检查: 每 30s 轮询 `/health`，连续 3 次失败触发自动重启

## 四层记忆系统

```
用户说了一句话
     │
     ▼
┌──────────────────────────────────────────────────────┐
│ Layer 1: 短期记忆 (Working Memory)                    │
│ ──────────────────────────────────────────────────── │
│ 最近 20 轮对话，完整消息                                    │
│ 存储: 内存数组 (per-session)                            │
│ 检索: 直接注入 LLM 上下文                                │
│ 生命周期: 当前语音会话                                     │
│ 容量: ~8K tokens                                     │
└──────────────────────────────────────────────────────┘
     │ 每轮对话后自动更新
     ▼
┌──────────────────────────────────────────────────────┐
│ Layer 2: 中期记忆 (Recent Summaries)                   │
│ ──────────────────────────────────────────────────── │
│ 近 7 天对话的自动摘要，滑动窗口                               │
│ 存储: ~/.ai-desktop-pet/memory/summaries/            │
│ 每文件: {date, summary, topics[], keyDecisions[]}      │
│ 检索: 语义相似度排序，top-3 注入                          │
│ 触发: 每日凌晨定时 + 对话空闲 >5min 时异步生成               │
│       不阻塞主对话流程，后台队列执行                          │
│ 容量: 7 个日摘要 + 1 个当前会话摘要                          │
└──────────────────────────────────────────────────────┘
     │ 日终自动摘要
     ▼
┌──────────────────────────────────────────────────────┐
│ Layer 3: 情景记忆 (Episodic Memory)                    │
│ ──────────────────────────────────────────────────── │
│ 完整的对话事件记录，包含时空上下文                              │
│ 存储: ~/.ai-desktop-pet/memory/episodes/             │
│ 每事件:                                                 │
│   {                                                    │
│     id, timestamp,                                     │
│     context: {                                         │
│       timeOfDay, dayOfWeek,                            │
│       activeFiles[],          // 对话时打开的文件         │
│       screenActivity,         // 粗略活动描述            │
│       prevEpisodeId           // 前一个情景              │
│     },                                                 │
│     emotionalState: {                                  │
│       userMood,              // 用户情绪                │
│       agentTone,             // Agent 回应语气          │
│       intensity              // 情绪强度 0-1            │
│     },                                                 │
│     content: {                                         │
│       topic,                                            │
│       userIntent,           // 用户意图分类             │
│       agentApproach,        // Agent 策略              │
│       outcome,              // 结果: resolved/pending/  │
│       decisions[],          // 做出的决定               │
│       keyQuote              // 关键引用                 │
│     },                                                 │
│     tags: [],                                          │
│     embedding: float[]      // 语义向量                 │
│   }                                                    │
│ 检索: 时间线查询 + 语义相似度 + 情绪状态匹配                 │
│ 用途: "上次我们聊到这个的时候..." 的上下文回忆               │
│ 情绪记忆: 复用语音情绪检测结果，记录不同场景下的用户情绪倾向   │
│          让 Agent 知道用户在特定环境下更易怒/更放松            │
└──────────────────────────────────────────────────────┘
     │ 关键信息提取
     ▼
┌──────────────────────────────────────────────────────┐
│ Layer 4: 长期记忆 (Fact Store)                         │
│ ──────────────────────────────────────────────────── │
│ 持久化事实知识库（已有 FactStore 增强）                     │
│ 存储: better-sqlite3                                  │
│ 每事实:                                                │
│   {                                                    │
│     id, fact, tags[],                                  │
│     source: { episodeId, conversationId },             │
│     confidence: 0-1,                                   │
│     createdAt, updatedAt,                              │
│     accessCount,                                       │
│     embedding: float[]                                 │
│   }                                                    │
│ 检索: 关键词 + 语义向量混合检索，top-5 注入                 │
│ 冲突解决: LLM 结构化 UPDATE/DELETE 指令 + 相似度去重          │
│          新事实与已有事实相似度 >0.85 → 合并更新而非重复插入    │
└──────────────────────────────────────────────────────┘
```

### Layer 5: 知识图谱 (Knowledge Graph) [North Star]

在四层记忆之上，构建实体关系网络，让 Sonder 从"记住事实"进化到"理解关系"。

**技术方案: LangGraph + FalkorDB**

```
实体节点: Person(Frank), Movie(星际穿越), Tech(Python), Project(Django迁移)
关系边:   [Frank] -liked→ [星际穿越]
          [Frank] -watched_in→ [2025年夏]
          [星际穿越] -directed_by→ [诺兰]
          [诺兰] -also_directed→ [盗梦空间]
          [Frank] -uses→ [Python]
          [Frank] -working_on→ [Django迁移bug]
```

**核心指标**: 图结构系统在多跳推理任务中比传统向量检索准确率提升高达 **37%**。

**落地路径**: 使用 `mem0-falkordb` 插件将记忆从扁平文本升级为结构化知识。LangGraph 构建图谱节点和边，FalkorDB 作为图数据库引擎，图遍历算法实现多跳推理查询。

多跳推理示例:
```
用户: "上次你推荐的那个电影叫什么来着？"
Sonder: 检索 [Frank] → (liked→星际穿越) → 回答
用户: "不是那部，导演拍了盗梦空间的那部"
Sonder: [Frank]→liked→? ←directed_by←[诺兰]→also_directed→[盗梦空间]
        推断: 去年夏天看过的 → 是《星际穿越》
        回答: "是《星际穿越》，你去年夏天看的。诺兰的另一部《盗梦空间》你也提过。"
```

**备选方案: ByteRover 启发式 Context Tree**

一种更轻量的方案——将记忆的整理和检索都交给 LLM 本身：

```
Context Tree (层次化上下文树):
  root/
    ├── facts/          # 事实知识 (Markdown 文件)
    │   ├── tech-stack.md
    │   └── preferences.md
    ├── episodes/       # 情景记忆
    │   └── 2026-06/
    └── reflections/    # 自我反思输出
        └── 2026-06-03.md

优势:
  - 无需外部向量数据库，数据存在本地可读 Markdown 文件
  - 亚毫秒级检索（文件系统级别）
  - 天然支持版本控制 (git diff 追踪记忆变化)
  - 数据隐私: 所有记忆在用户本地磁盘
```

### 记忆分层管理

```
┌────────────┐  热层 (Hot)    │ 当前会话 + 最近24h       │ 内存常驻，即时访问
├────────────┤  暖层 (Warm)   │ 7天摘要 + 30天情景       │ ChromaDB 语义检索，毫秒级
├────────────┤  冷层 (Cold)   │ 30天+历史归档            │ 磁盘 JSON/Markdown，按需加载
└────────────┘
```

**检索性能优化**:
- 中期摘要和情景记忆统一使用 ChromaDB 做语义检索，避免 SQLite + 手写 embedding 两套逻辑
- SQLite 仅存储结构化元数据（时间戳、标签、来源 ID），实际向量搜索走 ChromaDB
- 数据量达数万条时仍保持 <100ms 检索延迟

### Sleep Mode 触发条件细化

```
触发条件 (全部满足):
  1. 无任何交互 (语音/文字/鼠标移动) 超过 10 分钟
  2. 无后台任务正在执行 (无 LLM 请求、无 TTS 合成)
  3. 电池非低电量 (<15% 时不触发)
  4. 不在勿扰模式时段
  5. 当天 Sleep Mode 执行次数 <2 次

执行内容:
  1. 提取当日对话摘要 (LLM 总结，token 上限 4K)
  2. 检查是否有矛盾事实 → MERGE/UPDATE
  3. 高重要性事件写入情景记忆
  4. 更新用户情绪趋势 (可选，额外 +1K token)
  5. 清理过期录音文件

执行频率: 每天最多 2 次，最小间隔 4 小时
超时保护: 单次执行超时 60s 自动终止
```

### 自我反思 (Sleep Mode) [Phase 3]

系统空闲（>10分钟无交互）时自动进入反思模式。核心理念来自 ByteRover: **让 LLM 管理自己的记忆**。

```
Sleep Cycle:
  1. 回顾今日对话 → 提取关键决策、新事实、情感变化
  2. 整合碎片 — 将零散短时记忆整合为结构化 Markdown 知识单元
  3. 修正错误 — 从对话中的纠正和反馈中学习
  4. 冲突解决 — 使用 Mem0 的 UPDATE/MERGE 机制:
     - 新事实 vs 旧事实相似度 >0.85 → MERGE 合并
     - 新事实直接否定旧事实 → UPDATE 覆盖
     - 新事实补充旧事实 → APPEND 追加
  5. 更新知识图谱 — 添加新实体、修正关系、删除过期信息
  6. 生成"今日学习"日志 — 可 git diff 追踪记忆随时间的变化

反思输出示例:
  [今日反思 2026-06-03]
  了解了: Frank 在修 Django 4.2 迁移 bug，项目截止周五
  修正了: Frank 偏好 VS Code 而不是之前记录的 PyCharm
  冲突处理: MERGE "Frank使用Django" → 补充了版本号 4.2
  提示: Frank 连续 3 天提到同一个 bug，可能需要建议不同方案
```

### 外部记忆连接器 [North Star]

- **Mem0 集成**: 开源记忆层中间件，让 Sonder 的记忆在多个 LLM 应用间共享
- **第二大脑接入**: 连接 Obsidian/Notion/Logseq 等笔记系统，利用用户已有知识体系
- **外部 API**: 可扩展的连接器接口，支持接入新的记忆源

```
用户输入
  │
  ├─→ 短期记忆: 最近 20 轮直接注入 LLM
  ├─→ 中期记忆: 语义检索 top-3 摘要 → 注入
  ├─→ 情景记忆: 时间线 + 语义检索 top-3 相关情景 → 注入
  └─→ 长期记忆: 混合检索 top-5 事实 → 注入
  
LLM 收到上下文:
  [系统提示]
  [中期摘要] 以下是最近相关的对话摘要: ...
  [情景记忆] 以下是相关的历史对话情景: ...
  [长期事实] 以下是你已知的关于用户的事实: ...
  [短期记忆] 以下是当前对话: ...
  [用户最新输入]
```

### 情景记忆 vs 长期记忆

| 维度 | 情景记忆 | 长期记忆 |
|------|----------|----------|
| 记录什么 | 事件（时间/场景/情绪/过程/结果） | 事实（用户喜欢 X，项目用 Y） |
| 例子 | "6月3日下午，用户在修 Django 迁移 bug，很烦躁，Agent 建议 reset 数据库解决了" | "用户使用 Django 4.2" |
| 检索方式 | 时间线 + 语义 | 关键词 + 语义 |
| 用途 | "上次类似的场景…" | "据我所知…" |

## LLM 情感标签协议

LLM 在每个回复前输出情感标签（不显示给用户，仅用于 TTS 参数映射）：

```
格式: [emotion:标签]
标签: neutral | happy | sad | angry | worried | encouraging | funny | sarcastic

示例:
[emotion:sarcastic] 哇，这个代码写得真是"优雅"呢...你确定不是从 StackOverflow 复制粘贴的？
[emotion:worried] 你已经连续工作 4 个小时了，要不要休息一下？
[emotion:encouraging] 没关系，这个错误很常见，我们一起看看怎么解决...
[emotion:funny] 哈哈，这个 bug 的名字比你昨天取的那个还要离谱！
```

### 情感优先级规则

当用户情绪强烈时，Agent 回复情感应**优先匹配用户情绪（共情）**，而非默认中性：

| 用户情绪 | 优先级 | Agent 首选情感 | 说明 |
|----------|--------|----------------|------|
| 愤怒 | 最高 | worried → encouraging | 先共情安抚，再鼓励 |
| 悲伤 | 高 | sad → encouraging | 先共鸣，再引导 |
| 恐惧/焦虑 | 高 | worried → encouraging | 先理解，再缓解 |
| 快乐 | 中 | happy / funny | 共鸣，保持氛围 |
| 惊讶 | 中 | happy | 一起惊讶，共享体验 |
| 中性 | 低 | 根据对话内容自由选择 | 正常对话 |

### 情感 → CosyVoice 2 参数映射

CosyVoice2-0.5B 原生支持 7 种情感标签。LLM 输出映射到 CosyVoice 参数：

| LLM 标签 | CosyVoice 情感 | 语速倍率 | 音调偏移 | 说明 |
|----------|---------------|----------|----------|------|
| neutral | neutral | 1.0 | 0 | 默认中性 |
| happy | happy | 1.1 | +20Hz | 愉悦轻快 |
| sad | sad | 0.85 | -15Hz | 低沉缓慢 |
| angry | angry | 1.15 | +10Hz | 急促有力 |
| worried | sad | 0.9 | -8Hz | 担心忧虑 |
| encouraging | happy | 1.05 | +8Hz | 温柔鼓励 |
| funny | happy | 1.15 | +15Hz | 活泼俏皮 |
| sarcastic | neutral | 1.0 | +10Hz / -10Hz (语调起伏) | 阴阳怪气 |

### 原型验证策略

直接使用 CosyVoice2-0.5B 的 instruct 模式控制情感表达，无需经过 SSML 原型验证阶段。

## 用户情绪检测

在 STT 阶段分析用户语音的声学特征，判断情绪状态：

### 检测维度

| 维度 | 特征 | 意义 |
|------|------|------|
| 语速 | 字/秒 | 快→兴奋/焦虑，慢→疲惫/沉思 |
| 音调 | 基频均值 | 高→激动/紧张，低→冷静/沮丧 |
| 音量 | RMS 能量 | 大→愤怒/兴奋，小→疲惫/不确定 |
| 停顿 | 停顿频率与时长 | 多→犹豫/思考，少→流畅/自信 |
| 文字内容 | 语义辅助判断 | 结合用户说的话做双模态判断 |

### 情绪分类（6 维）

纯语音情感识别准确率约 70-80%，建议结合文字内容一起判断：

| 情绪 | 声学特征 | 文本特征 | Agent 应对策略 |
|------|----------|----------|--------------|
| 愤怒 | 快语速+高音调+大音量 | 负面词/感叹号/质问句 | 共情安抚 → 温和回应 |
| 快乐 | 快语速+高音调+大音量 | 正面词/笑声/轻松语气 | 共鸣保持氛围 |
| 悲伤 | 慢语速+低音调+小音量+多停顿 | 消极词/自我否定 | 温柔共鸣 → 鼓励引导 |
| 恐惧 | 极快语速+不规则停顿 | 担心词/假设最坏情况 | 安全确认 → 缓解焦虑 |
| 惊讶 | 极快语速+高音调+突然停顿 | 疑问词/感叹 | 一起惊喜，了解更多 |
| 中性 | 正常所有参数 | 陈述句/客观描述 | 根据内容自由选择 |

### 情感引擎置信度门控 (关键优化)

70-80% 的情绪识别准确率意味着每 4-5 次就有 1 次误判。加入置信度门控避免共情错位：

```
用户输入 → [声学: 强度0.6, 置信度0.75] → 未达阈值
         → 退回到认知引擎 (正常对话流程)

用户输入 → [声学: 强度0.85, 置信度0.82] → 通过阈值
         → 激活情感交互引擎

阈值设定:
  情绪强度 >0.7  AND  置信度 >0.8  → 启用情感引擎
  否则 → 保持认知引擎 (默认)
```

### 用户手动纠正

- 语音页面字幕气泡下方显示检测到的情绪标签（小字，灰色）
- 用户可点击标签 → 弹出正确情绪选择 → Sonder 记录修正
- 修正数据积累用于后续个性化微调（存储在情景记忆中，标注 `corrected: true`）

```
示例:
🤖 Agent 回复...
💡 检测情绪: 你(悲伤) → Agent(encouraging)
      └─ [✎ 纠正] → 选 开心 中性 愤怒...
```

### 多模态情绪融合

```
用户说: "哈哈这个bug又回来了" (文本含「哈哈」→ 正面)
       但语速极快、音调尖锐 (声学 → 焦虑)
─────────────────────────────────────────
声学强度: 0.78, 文本正面度: 0.65
融合判断: 焦虑/烦躁 (权重: 声学 60% + 文本 40%)
置信度: 0.72 (声学和文本信号冲突，置信度降低)
→ 置信度 <0.8，退回认知引擎，避免跟着"哈哈"
→ 用 worried + encouraging 先共情
```

### 实现

- 浏览器 Web Audio API AnalyserNode 提取声学特征（基频/能量/过零率）
- 启发式规则分类器（主力）+ 可选 ONNX 情绪模型（增强）
- 结果注入 LLM 上下文: `[用户情绪: 焦虑]用户说: "这个 bug 修了一天了"`
- 情景记忆复用情绪检测结果，构建用户的"情绪画像"

### 情绪持续性跟踪 (关键优化)

单次情绪检测容易受噪声干扰。结合历史上下文做滑动窗口平滑：

```
情景记忆中的情绪曲线:
  episodes/2026-06-03.json:
  {
    "emotionCurve": [
      { "ts": "14:30:05", "emotion": "中性", "intensity": 0.3 },
      { "ts": "14:30:18", "emotion": "焦虑", "intensity": 0.65 },
      { "ts": "14:31:02", "emotion": "愤怒", "intensity": 0.82 },  ← 连续累积
      { "ts": "14:31:45", "emotion": "愤怒", "intensity": 0.91 }   ← 情绪升级
    ]
  }

滑动窗口计算:
  窗口大小: 最近 3 轮对话
  平均情绪 = Σ(emotion_weight × intensity) / 3
  
  示例:
  轮1: 焦虑 (0.65) + 轮2: 愤怒 (0.82) + 轮3: 愤怒 (0.91)
  → 平均强度 0.79，主导情绪: 愤怒
  → 触发情感交互引擎 → Agent 用 worried + encouraging 共情

效果: 用户说两次"烦死了"，情绪强度从单次 0.65 累加到窗口平均 0.79
      避免因单次误判（如随口一句"好烦"）就切换引擎
```

## 多模态特征提取与融合

### 文字模态特征

| 特征类型 | 提取内容 | 用途 |
|----------|----------|------|
| 显式情感 | 情绪词（"开心""烦死了"）、强度词（"非常""有点"） | 直接推断用户情绪标签 |
| 隐式情感 | 否定结构、反问句、重复标点（"！！！"） | 识别潜在负面情绪 |
| 话题实体 | 人名、地点、产品名、领域术语 | 构建用户兴趣图谱 |
| 句式风格 | 正式/口语、长短句比例、礼貌用语频率 | 匹配 Agent 回复风格 |
| 意图变化 | 连续多轮中意图跳跃程度（"聊猫"→"问编程"） | 判断用户专注度 |

技术: `text2emotion` 或微调 BERT-tiny 做情感分类；LLM (deepseek-v4-flash) 做实体抽取和话题分类。

### 语音模态特征

| 特征类型 | 提取内容 | 用途 |
|----------|----------|------|
| 声学情感 | 基频(F0)、能量、语速、共振峰 | 识别愤怒、焦虑、兴奋 |
| 副语言 | 笑声、叹息、填充词（"呃""嗯"） | 判断犹豫、放松、尴尬 |
| 语音质量 | 音量稳定性、声音颤抖、气息声 | 检测疲劳、生病或紧张 |
| 韵律结构 | 句子边界停顿、重音位置 | 理解用户强调重点 |
| 交互行为 | 打断频率、响应延迟 | 评估用户耐心、投入度 |

技术: Wav2Vec2 微调版或 MSP-Podcast 预训练模型。VAD 提取说话段落声学特征，计算均值/方差作为输入。

### 多模态融合策略

| 融合层级 | 方法 | 适用场景 |
|----------|------|----------|
| 特征级 | 文本 embedding + 声学特征拼接 → 分类器 | 实时情绪识别 |
| 决策级 | 文本和语音分别预测 → 加权投票 | 可解释性要求高，允许单模态失败 |
| 模型级 | 多模态 Transformer (CLAP/WavLM+text) | 最高准确率，但计算量大 |

**初期采用决策级融合**: 文本情绪权重 60%，语音情绪权重 40%（语音更易受环境噪声影响）。后期训练线性加权模型，根据各模态置信度动态调整。

### 多模态模型选型 (Phase 3-4)

| 模态 | 推荐模型 | 部署 | 延迟 | 说明 |
|------|----------|------|------|------|
| 文字情感 | text2emotion (轻量) 或 bge-micro-v2 + 分类头 | 本地 Python | <10ms | 纯 CPU 推理，无需 GPU |
| 语音情感 | Wav2Vec2-base 微调版 或 HuBERT + 情绪分类头 | ONNX Runtime (本地) | <50ms | CPU 实时，可导出 ONNX 加速 |
| 文本实体抽取 | deepseek-v4-flash (云端 LLM) | API 调用 | ~200ms | 成本极低，准确率高 |
| 声学特征提取 | librosa (MFCC, 基频, 能量, 过零率) | 本地 Python | <5ms | 纯 CPU 计算，无模型加载 |
| 文本话题分类 | BERTopic 或 LDA | 本地 Python | <100ms | 离线批处理，每周运行一次 |

``` 
特征提取管线:
  [浏览器] Web Audio API → PCM 音频流
      │
      ├─→ [本地] librosa 声学特征 (F0/能量/MFCC/语速) <5ms
      ├─→ [本地] Wav2Vec2 ONNX 语音情感 <50ms
      └─→ [本地] text2emotion 文字情感 <10ms
              │
              ▼
         [Python] 决策级融合 (60/40 → 动态权重)
              │
              ▼
         [注入] LLM 上下文 + 用户画像更新
```

## 用户理解系统: 五层进化

```
L5 情绪健康    │ 长期趋势、心理预警          │ 情绪跟踪 + 阈值告警
L4 人格建模    │ 性格倾向、决策风格           │ 心理量表映射 + 行为统计
L3 行为预测    │ 预判下一步动作               │ 序列预测模型
L2 动态偏好    │ 近期兴趣、话题趋势           │ 时序分析 + 分类模型
L1 事实记忆    │ 姓名、城市、宠物等           │ 向量检索 + 用户画像
```

当 Sonder 达到 L3 以上时，用户明显感受到"它越来越懂我了"——不是靠存储更多事实，而是靠持续推断、预测和主动关怀。

### L1-L2: 从静态标签到动态权重

现状 `user_profile.json` 存储静态事实。升级为每个属性携带置信度和时效性：

```json
{
  "attribute": "喜欢猫",
  "value": true,
  "confidence": 0.9,
  "last_updated": "2026-06-01",
  "half_life_days": 90,
  "source_rounds": ["对话1", "对话3"]
}
```

**规则**:
- 新信息与旧信息一致 → confidence += 0.05 (上限 0.95)
- 不一致 → 降权或触发冲突解决
- 时效性强的属性（"心情""正在做的项目"）→ 短半衰期，长时间未更新自动降权
- 每个特征独立半衰期: "当前情绪" 30分钟、"本周兴趣" 7天、"长期性格" 30天以上

**指数衰减**:
```
weight_new = weight_old × 2^(-delta_t / half_life)
```
Sleep Mode 中扫描所有特征，低于阈值的归档或删除。

### L2: 分层画像——表层偏好 vs 深层人格

| 层级 | 内容 | 生命周期 | 推断方式 |
|------|------|----------|----------|
| 表层偏好 | 喜欢的音乐/电影/食物 | 30天内快速变化 | 直接询问 + 行为统计 |
| 深层人格 | 内向/外向、理性/感性、价值观 | 需≥100轮有效对话 | Big Five 简版心理模型映射 |

**行为→人格映射示例**:
- "经常问怎么办" → 焦虑倾向高 (neuroticism↑)
- "主动开启新话题" → 开放度高 (openness↑)
- "回复简短，不爱寒暄" → 内向倾向 (extraversion↓)

每周更新一次人格模型，使用 Gaussian Process 输出每个维度的概率分布。

### L3: 主动推理与意图预测

**行为预测**:
```
用户连续一周每晚 10 点说"帮我定个明天 7 点的闹钟"
→ Sonder 在 21:55 主动: "要帮你定明天的闹钟吗？"
```

**话题趋势**:
- 对历史对话做主题建模 (LDA/BERTopic)，输出每周热门话题
- 用户再次问及 → Agent 主动: "上次你说想找猫粮，我帮你留意了几款..."

**技术**: 记录行为时间序列 `(timestamp, action_type, params)` → LSTM/Transformer 预测下一步概率 → >0.7 时主动发起

**冷启动策略** (关键):

用户初期 <100 轮对话时无法训练可靠模型。分层启动：

```
阶段 1: 规则引擎 (< 50 轮)
  - "连续 3 天同一时段问天气" → 主动推送
  - "连续 5 次在 22:00 后说'累死了'" → 关怀提醒
  - "本周第 3 次问同一个技术问题" → 主动整理资料
  - 仅使用硬编码规则，不启用模型预测

阶段 2: 混合模式 (50-200 轮)
  - 规则引擎 + 简单统计模型 (移动平均/频率计数)
  - 规则触发 → 模型验证 → 双重确认后推送

阶段 3: 模型主导 (>200 轮)
  - LSTM/Transformer 预测为主
  - 可选迁移学习: 公开行为序列数据集预训练 → 用户数据微调
  - 规则作为安全兜底 (防止模型误判过度推送)
```

**主动推送频率限制**:
- 每天最多 2 次主动推送
- 用户连续 3 次忽略 → 当天暂停推送
- 勿扰时段(22:00-08:00)永不来推送

**主动推送示例**:
```
用户连续三天晚上 10 点语音说"好累啊" (语速慢、能量低、叹息多)
→ Sonder 在 21:50: "你最近几天晚上都显得很疲惫，要不要我提醒你早点休息？"
```

### L4-L5: 长期情绪模式与心理健康感知

**情绪趋势分析**: 用户情绪检测结果按时间戳存入时序数据库，每周生成情绪报告。

**三级预警协议**:

```
┌─────── 级别 1: 轻度关怀 ───────────────────────────────┐
│ 触发: 负面情绪比例 >40% 持续 1 周                          │
│       或检测到周期性压力模式 ("每周一早上焦虑")              │
│ 行动: 主动询问 "你最近似乎有些低落，需要我陪你聊聊吗？"      │
│       推送放松练习或正念音频 (可选)                         │
│       语气: gentle + encouraging                          │
│ 记录: 写入情景记忆，标注 level_1                           │
├─────── 级别 2: 中度关注 ───────────────────────────────┤
│ 触发: 负面情绪比例 >60% 持续 2 周                          │
│       或频繁使用 "累死了""撑不住了""没希望了" 等词          │
│ 行动: 推送正念练习/呼吸指导/放松音频                        │
│       "我注意到你最近压力似乎很大。如果愿意，可以和我聊聊，   │
│        也可以试试我推荐的这个呼吸练习。"                     │
│       语气: worried + gentle                              │
│       通知: 不主动外发，仅在情景记忆中记录                   │
├─────── 级别 3: 严重预警 ───────────────────────────────┤
│ 触发: 检测到自伤/自杀倾向关键词                             │
│       "不想活了""活着没意思""结束这一切"                    │
│ 行动: 1. 立即停止当前对话                                  │
│       2. 显示心理援助热线:                                  │
│          "我听到了一些让我很担心的话。请记得，寻求帮助是      │
│           勇敢的表现。以下是你可以联系的资源："              │
│          全国心理援助热线: 400-161-9995                     │
│          生命热线: 400-821-1215                             │
│       3. 不尝试自行心理干预                                 │
│       4. 后续对话中不主动提及此事，等待用户自己开启           │
│ 记录: 写入加密日志，仅本地存储                              │
│ 隐私: 用户可选择将预警信息加密匿名后用于模型改进 [默认关闭]   │
└──────────────────────────────────────────────────────────┘
```

**重要安全边界**:
- Sonder 提供陪伴和关怀，**不是医疗诊断工具**
- 任何预警不自动发送给第三方（家人/医生）
- 级别 3 仅在用户明确提到自伤时才触发，不做推断
- 预警日志加密存储，不可被其他应用读取

### 跨会话持续学习

**每周知识整合** (Sleep Mode 子任务):
1. 分析过去 7 天对话 → 提取新用户洞察
2. 更新用户画像 (置信度 + 时效性)
3. 合并相似事实 ("喜欢喝美式" + "每天喝咖啡" → "每日饮用美式")
4. 移除过时/已覆盖信息

**知识蒸馏** [North Star]:
- 长期记忆达数千条时，训练轻量级用户偏好预测器 (100M 参数 MLP)
- 直接根据当前对话上下文预测潜在需求，避免每次都检索全量记忆

### 对话风格个性化迁移

**学习用户表达风格**:
| 维度 | 来源 | Agent 调整 |
|------|------|------------|
| 长度偏好 | 历史回复字数统计 | 动态调整 max_tokens，"回复控制在 X 字以内" |
| 语言风格 | 分类模型识别 | 匹配情感标签 (轻松幽默 / 专业严肃 / 温柔鼓励) |

```

### 自然反馈词 (Backchanneling) [Phase 2]

在半双工模式下，Backchanneling 存在设计冲突：系统在 LISTENING 状态时，用户正在说话，播放反馈声音会打断用户。

**初期方案**: 视觉反馈为主，音频反馈为辅

| 模式 | 反馈方式 | 实现 |
|------|----------|------|
| 视觉反馈 | 字幕区显示 "嗯" "然后呢？" 文字，仅 800ms 短暂出现 | 检测停顿 >800ms + 情绪触发 |
| 音频反馈 | 仅在用户停顿间隙以极小音量播放（10% 系统音量） | VAD 确认静默 >1.5s 才播放 |
| 打断保护 | 如果反馈音频播放中用户再次开口 → 立即停止反馈音频 | VAD 优先级高于反馈 |

**演进方向** [North Star]: 全双工模式下，系统在用户说话间隙以极小音量插入反馈，更自然但实现复杂。

**初期简化**: 可先只保留视觉反馈（字幕"嗯"），完全不做音频 Backchanneling，降低实现复杂度。

| 用户情绪 | 反馈词 | 触发时机 |
|----------|--------|----------|
| 中性叙述 | "嗯"、"嗯嗯" | 用户停顿 >1.5s |
| 兴奋分享 | "哇"、"真的吗" | 用户语气上扬 + 停顿 |
| 沮丧抱怨 | "我懂..." | 用户叹气或长停顿后 |
| 困惑 | "嗯？" | 用户语气不确定 + 停顿 |
| 长篇叙述 | "然后呢？" | 用户讲到一半长停顿 |

实现: 轻量级规则引擎（VAD 检测停顿 >1.5s + 用户上句情绪 → 触发反馈词），不依赖 LLM，延迟 <100ms。

### 动态人格模型 [North Star]

Sonder 的人格不是固定的，而是在与用户的互动中逐渐塑造：

```
人格维度（0-1 连续谱）:
  warmth:     冷 0.0 ←→ 1.0 热     // 亲和力
  humor:      正 0.0 ←→ 1.0 皮     // 幽默感
  directness: 婉 0.0 ←→ 1.0 直     // 直接性
  curiosity:  被 0.0 ←→ 1.0 主     // 好奇心
  empathy:    理 0.0 ←→ 1.0 感     // 共情度

演化规则:
  - 用户频繁用幽默回应 → humor 维 +0.02
  - 用户表达需要直接答案 → directness +0.03
  - 用户情绪低落时 Sonder 温暖回应获得感谢 → warmth +0.02, empathy +0.02
  - 每 50 轮对话自动评估一次人格漂移趋势
```

## 实时交互增强 [North Star]

### 轮次预测 (Turn Prediction) [North Star]

从"停顿检测"进化到"预测结束点"。参考字节跳动 Seeduplex 全双工技术的量化指标：

```
当前: 检测到静默 >2s → 认为用户说完了
升级目标 (参考 Seeduplex 指标):
  - 判停延迟降低约 250ms
  - AI 抢话比例减少 40%
  - 打断响应延迟额外缩短约 300ms

轻量级实现路径:
  1. VAD 检测用户说话停顿 (>800ms)
  2. 句法完整性判断 — 轻量模型判断句子是否语法完整
  3. 韵律特征 — 句末音调下降/拖长
  4. 语义完整性 — LLM 判断是否表达完整意图
  5. 历史模式 — 学习用户通常的停顿模式和说话节奏

预测模型: 
  P(turn_end | 句法完整 + 音调下降 + 语义完整 + 停顿>800ms) ≈ 95%
  → 在用户说完 ~200ms 内 Sonder 无缝接话
```

### GPT-4o Realtime API 备选

当本地 CosyVoice + 云端 LLM 的管线出现质量或延迟问题时：

```
主方案: CosyVoice2-0.5B (本地) + Claude/DeepSeek (云端)
备选方案: OpenAI GPT-4o Realtime API (一站式语音)
  - 优势: 端到端 <500ms 延迟，原生语音理解+生成
  - 劣势: 费用高，隐私风险
  - 触发: 用户手动切换 或 主方案连续 5 次延迟 >5s
```

## CosyVoice 2 服务

### 部署

```
Conda 环境: cosyvoice
Python: 3.10+
模型: CosyVoice2-0.5B (ModelScope)
服务: Flask HTTP 监听 127.0.0.1:5001
启动: Electron main process spawn Python 子进程
```

### API

```
POST /tts/generate
  Body: {
    "text": "要合成的文本",
    "emotion": "happy",
    "voice_id": "default_female",  // 预训练音色 ID
    "speed": 1.1,
    "stream": true                 // 流式返回 WAV chunks
  }
  Response (stream): multipart WAV chunks
  Response (non-stream): audio/wav

GET /tts/voices
  Response: {
    "voices": [
      {"id": "default_female", "name": "默认女声", "tags": ["warm", "clear"]},
      {"id": "default_male", "name": "默认男声", "tags": ["deep", "calm"]},
      ...
    ]
  }

POST /tts/clone
  Body: multipart/form-data { audio: file, name: string }
  Response: { "voice_id": "cloned_xxx", "status": "ready" }

GET /health
  Response: { "status": "ok", "model": "CosyVoice2-0.5B", "loaded_voices": 5 }
```

### 启动优化

- Electron 启动时即 spawn Python 服务，模型常驻内存，后续请求即时响应
- 模型加载 5-10s，期间语音页显示"语音引擎加载中..."
- 保持 HTTP Keep-Alive 长连接，避免每次请求重新握手

### 流式合成 (关键优化)

CosyVoice2 支持流式推理——边生成边播放，不解码完整音频后才开始输出：

```
传统方式: LLM输出全文 → TTS合成全文(2s) → 开始播放
流式方式: LLM 输出第一句 → TTS合成第一块(~300ms) → 开始播放
          同时 LLM 生成第二句 → TTS 合成第二块 → 无缝衔接播放

效果: 感知延迟从 1-2s 降至 ~300ms (首个音频块)
      长回复多句不会累积延迟，用户几乎无等待感
```

**分句策略**:
- LLM 流式输出时按标点分句（。！？\n）
- 每句完成立即送入 CosyVoice 流式合成
- 首包音频 300-500ms 内输出，长回复不累积延迟

**播放队列与打断优先级**:
```
Agent 播放中，新 TTS 请求到达:
  ┌──────────────────────────────────────┐
  │ 谁触发的？                            │
  ├──────────┬───────────────────────────┤
  │ 用户新消息 │ 当前播放的旧回复被替换       │
  │ (连续提问)│ 新回复首包到达 → 切换播放    │
  ├──────────┼───────────────────────────┤
  │ 用户打断   │ 立即清空队列 + 取消所有       │
  │ (Esc/语音)│ TTS 生成任务 → LISTENING   │
  └──────────┴───────────────────────────┘
```
- 队列策略: 队列仅缓存当前回复的分句（最多 5 句），新回复到达时旧队列被替换
- 打断保护: 播放 <500ms 的句子不可被打断（防止误触发）

**录音存储管理**:
```
存储路径: ~/.sonder/recordings/YYYY-MM-DD/
文件命名: HH-MM-SS-turn{N}.opus
生命周期: 默认保留 7 天，可配置为 不保存/7天/30天
清理策略: 每次启动时检查，超期自动删除
UI 开关:  设置 → 语音 → 录音保存策略 (下拉选择)
隐私:     录音仅存本地，不上传。对话回放从本地加载
```

### 情感/音色预加载

- 常用情感（neutral/happy/gentle）的模型权重在启动时预加载到内存
- 切换情感标签时无需重新加载模型，仅切换推理参数
- 首次使用冷门情感（如 sarcastic）时可能有额外 500ms 加载延迟
- 音色切换同理：预加载当前 Agent 音色，切换时参数切换 <100ms

### 并发控制

- TTS 请求队列（最大并发 1），避免同时多个合成任务导致显存溢出
- 新请求到达时：如果正在合成 → 取消当前任务 → 开始新任务（打断场景）
- 队列满时返回 429 Too Many Requests，前端检测后等待重试

### 音频格式

- CosyVoice 原始输出: PCM 16kHz 16bit 单声道 → WAV
- 传输前转换为 Opus 编码（~10x 压缩），1 分钟语音 <500KB
- 前端接收 Opus → Web Audio API decodeAudioData() → 播放
- 录音存储: Opus 格式直写磁盘

### 进程生命周期

```
Electron 启动
  → spawn python cosyvoice-server.py
  → 等待 /health 返回 ok (超时 60s，模型加载中: 5-10s)
  → 就绪

语音页打开
  → 检查服务健康
  → 如挂了自动重启（最多 2 次）

Electron 退出
  → app.on('will-quit') → SIGTERM → Python 进程树清理 (tree-kill)
  → 超时 5s 未退出 → SIGKILL
```

### 主动状态沟通

Sonder 不静默失败，而是主动告知用户当前状态：

- 文字提示: 字幕区显示 "⚠️ 语音服务暂时不可用，已切换为文字模式"

## 降级策略

### 降级触发与恢复

| 条件 | 动作 | 恢复机制 |
|------|------|----------|
| CosyVoice 连续 3 次超时(>10s)/错误 | 纯文字模式 + 提示"语音暂时不可用" | 每 5 分钟重试 `/health`，成功自动切回 |
| CosyVoice 恢复可用 | 从纯文字 → CosyVoice | 下句切换，平滑无感 |

### TTS 降级链

```

  │
  ├─ OK → 使用 CosyVoice ✓
  │

       │
       ▼

  CosyVoice 连续 3 次超时/错误

### 其他降级

| 场景 | 降级方案 |
|------|------|
| STT 不支持 | 自动退化为纯文字输入，语音页面仍可手动输入文字 |
| 用户情绪检测失败 | 跳过情绪标签，使用 [emotion:neutral] 正常对话 |
| 情景记忆检索超时(>3s) | 只注入短期 + 中期 + 长期，跳过情景层 |

| 长期 FactStore 检索失败 | 回退关键词匹配，跳过语义检索 |
| 麦克风权限被拒 | 自动切换到按键模式，提示用户授权 |

## 勿扰模式

- **触发**: 托盘菜单 / 语音页面按钮 / `Ctrl+M`
- **行为**: Agent 仅在聊天窗口文字回复，不出声
- **指示**: 托盘图标叠加 🔇 标记，语音页面状态显示 🔇
- **自动**: 可配置时间段（如 22:00-08:00 自动静音）
- **持久化**: 静音状态写入 config
- **"仅文字回复"模式**: 即使开启语音输入，Agent 也只输出文字不播放声音（比静音更彻底，连 TTS 合成都不触发）
- **快速切换**: 托盘图标右键 → "切换语音/仅文字模式"

## 敏感内容过滤

语音对话中检测到以下模式时暂停录音 + 不保存该片段：

- 连续数字串（≥8位，疑似银行卡/身份证）
- 密码关键词（"密码是"/"password"）
- 正则匹配: `/^(?=.*\d)(?=.*[a-zA-Z]).{8,}$/`（疑似密码）
- 匹配到敏感内容 → 播放短促警告音 → 气泡提示"检测到敏感内容，此片段未录音"
